    31. Design a Leaderboard for Fantasy Teams

    üìú Problem Statement

    Build an in-memory leaderboard for a fantasy-sports application.

    • Each user creates exactly one team
    • Each team contains one or more players
    • Players earn positive or negative points over time
    • A user‚Äôs score is the sum of scores of all players in their team
    • The system must support Top-K leaderboard queries

    üß† Rules & Constraints

    Team & Player Rules

    • Every user has exactly one team
    • A player can belong to multiple users‚Äô teams
    • Each user starts with score = 0
    • Player score updates are deltas (positive or negative)

    Leaderboard Ordering

    1. User score (descending)
    2. User ID (lexicographically ascending) as tie-breaker

    üß© APIs to Implement

    1Ô∏è‚É£ addUser

    void addUser(String userId, List<String> playerIds)
    • Registers a new user with their team
    • userId is unique and non-blank
    • playerIds contains at least one valid player
    • If players already have scores, user score should reflect them immediately

    2Ô∏è‚É£ addScore

    void addScore(String playerId, int score)
    • Updates a player‚Äôs cumulative score by a delta
    • All users containing that player must have their scores updated
    • Leaderboard ordering must stay consistent

    3Ô∏è‚É£ getTopK

    List<String> getTopK(int k)

    • Returns top k users sorted by:

      • Score (descending)
      • User ID (ascending)
      • If k > total users ‚Üí return all users

    üìò Example Walkthrough

    addUser("uA", ["p1", "p2"])
addUser("uB", ["p2"])

getTopK(2)
// ["uA", "uB"]  (both scores = 0)

addScore("p2", 10)
// uA = 10, uB = 10

getTopK(2)
// ["uA", "uB"]  (tie ‚Üí lex order)

addScore("p1", 3)
// uA = 13, uB = 10

getTopK(1)
// ["uA"]

addScore("p2", -5)
// uA = 8, uB = 5

getTopK(5)
// ["uA", "uB"]

    üèóÔ∏è Design Overview

    Core Entities

    Entity Responsibility
    Player Holds player score & notifies users
    User Maintains total team score
    Leaderboard Maintains sorted ranking

    üß† Design Pattern Used

    ‚úÖ Observer Pattern

    • User observes Player

    • When a player‚Äôs score changes:

      • All users owning that player are notified
      • User scores update automatically
      • Ensures efficient many-to-many updates

    üíª Java Implementation

    import java.util.*;

public class Leaderboard {

    private Map<String, Player> players = new HashMap<>();
    private Map<String, User> users = new HashMap<>();

    private TreeSet<User> leaderboard;

    public Leaderboard() {
        leaderboard = new TreeSet<>((a, b) -> {
            if (a.totalScore != b.totalScore) {
                return b.totalScore - a.totalScore; // score DESC
            }
            return a.userId.compareTo(b.userId); // lex ASC
        });
    }

    // ---------------- addUser ----------------
    public void addUser(String userId, List<String> playerIds) {

        User user = new User(userId);
        users.put(userId, user);

        for (String pid : playerIds) {
            Player p = players.computeIfAbsent(pid, k -> new Player(pid));
            p.addObserver(user);
            user.totalScore += p.score; // reflect existing score
        }

        leaderboard.add(user);
    }

    // ---------------- addScore ----------------
    public void addScore(String playerId, int delta) {

        Player p = players.computeIfAbsent(playerId, k -> new Player(playerId));

        // Remove affected users before score change
        for (PlayerObserver obs : p.observers) {
            leaderboard.remove((User) obs);
        }

        p.addScore(delta); // notify users

        // Reinsert updated users
        for (PlayerObserver obs : p.observers) {
            leaderboard.add((User) obs);
        }
    }

    // ---------------- getTopK ----------------
    public List<String> getTopK(int k) {
        List<String> result = new ArrayList<>();
        int count = 0;

        for (User u : leaderboard) {
            result.add(u.userId);
            count++;
            if (count == k) break;
        }
        return result;
    }
}

// ---------- SUPPORTING CLASSES ----------

class Player {
    String playerId;
    int score = 0;
    List<PlayerObserver> observers = new ArrayList<>();

    Player(String playerId) {
        this.playerId = playerId;
    }

    void addObserver(PlayerObserver observer) {
        observers.add(observer);
    }

    void addScore(int delta) {
        score += delta;
        notifyObservers(delta);
    }

    private void notifyObservers(int delta) {
        for (PlayerObserver obs : observers) {
            obs.onPlayerScoreChanged(delta);
        }
    }
}

interface PlayerObserver {
    void onPlayerScoreChanged(int delta);
}

class User implements PlayerObserver {
    String userId;
    int totalScore = 0;

    User(String userId) {
        this.userId = userId;
    }

    @Override
    public void onPlayerScoreChanged(int delta) {
        totalScore += delta;
    }
}

    ‚ö†Ô∏è Edge Cases Handled

    • Player added after scoring
    • Negative score updates
    • Tie-breaking by user ID
    • k larger than user count
    • Many-to-many player-user mapping

    ‚è±Ô∏è Time Complexity

    Operation Complexity
    addUser O(p log u)
    addScore O(u log u)
    getTopK O(k)

    p = players per user, u = users affected

    ‚úÖ Why This Is Interview-Ready

    • Clean Observer pattern usage
    • Deterministic ranking with TreeSet
    • Efficient updates without full recomputation
    • Easily extensible (Top-N per league, snapshots, persistence)
